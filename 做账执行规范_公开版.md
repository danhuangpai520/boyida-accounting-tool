# 钢材磅单 / 送货单 / 调拨单 → Excel 做账强制执行规范

## 0. 强制执行原则

本文件是唯一执行规范。任何执行者，包括人工执行者和 AI 执行者，拿到本文件后必须按本文档执行，不得另行改写需求、缩小范围或自行发明替代流程。

执行过程只允许提效，不得改变最终需求和验收标准：

- 最终 Excel 仍必须可做账、可核对、可追溯原图。
- 字段规则、重复图片规则、备注规则、Excel 样式和原图追溯要求不得因流程优化而降低。
- 开始处理前，必须先检查是否已有 OCR JSON、结构化 CSV / JSON 和最终 Excel，再按本文档判断是否重跑。
- 有可用 OCR JSON 时，必须基于已有结果修整，不得直接整批重跑 OCR；但“可用”必须满足当前批次所有图片都有非空 OCR 文本、没有 `error`、没有混入旧图片记录。
- 软件默认必须在 EXE 所在位置创建 `YYYY-MM-DD_保谊达做账表` 批次总目录，所有输入、输出和留档文件都放在该总目录内，不得把多个文件夹散落在桌面。
- 软件执行区必须显示当前任务进度、百分比、预计剩余时间和滚动终端日志；中间流程区下方是唯一主日志区，不再保留底部重复日志框。
- 软件使用无边框高端界面时，必须保留窗口边缘/角落拖拽缩放能力；默认窗口高度必须收紧，三栏布局不得在底部留下大块无意义空白，GPS 时间不得被 OCR 面板或内部数据块遮挡，中间日志字号必须可读。
- 每次重新打包 `做账执行工具.exe`，必须自动在 `项目资料\版本备份` 留下可回退版本，至少包含 EXE、当前源码、执行规范、使用说明和 SHA256 校验信息；版本备份不得包含明文 API Key。
- 本地 5090 主要用于提升 OCR 批处理速度、TensorRT / FP16 推理和并发吞吐，不允许让模型猜测做账字段。
- 疑问行必须单独复核；整批替换 OCR 引擎前必须先做抽样对比。

## 1. 任务目标

将一个批次文件夹中的钢材磅单、送货单、发货单、调拨单照片整理为可直接做账、可直接核对、可直接交付的 Excel。

固定交付文件：

`运输录入_<日期范围>.xlsx`

固定保留页签：

- `运输录入`
- `重复图片`

固定删除页签：

- 空的 `未匹配驾驶员`
- 已经完整回填到主表 `备注` 的 `识别疑问`

## 2. 执行原则

- 每次执行按当前文件夹视为一个独立批次，不与历史批次合并。
- 同一批次 OCR 只跑一轮，后续基于中间 JSON、Excel 和本地原图离线修整。
- 涉及结算字段，无法确认时留空并备注，不允许猜测补全。
- 能进入主表的问题必须进入主表 `备注`，不只放在辅助页。
- 重复图片宁可少删，不可误删。
- 原图列默认使用外部原图链接，不允许用低清二次压缩缩略图替代。
- 最终 Excel 必须满足固定列结构、固定样式、可筛选、可核对、可追溯原图。

如果图片来自微信文件传输助手，必须优先用微信界面直接保存图片，不得把时间浪费在读取或解密微信缓存文件上。

软件默认目录结构：


- 不读取 `xwechat_files` 缓存图片作为默认流程；微信缓存存在封装或加密，除非用户明确要求研究缓存，否则不得走解密路线。

## 6. 字段规则

### 6.1 日期

日期格式固定：

`YYYY/M/D`

示例：

- `2026/5/1`
- `2026/5/11`

可用日期依据：

- 单据上明确日期。
- 毛重时间或皮重时间中的日期。
- 磅单编号中带日期时，可还原日期。

编号日期示例：

- `S260507020` → `2026/5/7`
- `B202605080086` → `2026/5/8`

处理规则：

- 日期来源一致时直接填写。
- 编号日期、毛重时间、皮重时间存在冲突时，填写可确认日期，并备注 `日期冲突需核对`。
- 不凭文件创建时间猜测单据日期，除非用户明确要求。

### 6.2 车号

- 以单据识别结果为基础。
- OCR 与固定车号表只有轻微误差，且能唯一可靠匹配时，可修正为标准车号。
- 无法确认时留空，并备注 `缺少车号` 或 `车号模糊`。

### 6.3 驾驶员

- 根据 `车号` 在固定对照表中匹配。
- 匹配不到则留空，并备注 `未匹配到驾驶员`。

### 6.4 重量

- 录入实际结算重量。
- 原始单位为 kg 时换算为吨。
- 保留两位小数。
- 单元格中只写数字，不带单位文字。

优先读取：

- `结算重量`
- `净重`
- 明确写在结算位置附近的重量

处理规则：

- `52010 kg` 录入为 `52.01`。
- 不把毛重、皮重当作结算重量。
- 不把车号、编号、日期、时间中的数字当作重量。
- OCR 将字段拆成多行时，结合 `净重`、`结算重量`、`kg` 等上下文判断。
- 不再按固定阈值自动备注 `重量异常,需核对`。
- 只有重量本身识别不清、位置冲突或多处重量无法唯一判断时，才备注 `重量模糊` 或 `重量冲突需核对`。

### 6.5 货物名称

归一规则：

- `合金123` → `钢坯`
- `合金XXX` / `合金xxx` → `钢坯`
- `合金坯123`、`合合坯123` 等能确认属于合金类单据时 → `钢坯`
- `卷子板` 保持 `卷子板`
- `卷予板`、`卷子领`、`卷子饭`、`签予领` 等能结合原图确认属于卷子板时 → `卷子板`
- `钠坯`、`销坯`、`销址` 等能结合原图确认属于钢坯时 → `钢坯`
- `钠渣` 等能结合原图确认属于钢渣时 → `钢渣`
- `钢渣`、`废钢`、`料头` 等其他货物按原文填写

### 6.6 装货地 / 收货地

当货物名称为 `钢渣`：

- `装货地 = 保利`
- `收货地 = 新登江丰`

当货物名称为 `卷子板`：

- `装货地 = 友谊`
- `收货地 = 保利`

当货物名称为 `钢坯`：

优先按单据顶部标题和整体打印版式判断收货地：

- `独立过磅单标题版式`：顶部以独立的 `过磅单` 四字作为标题，`浙江钧延贸易有限公司` 与标题分开显示，且整体打印版式与新登江丰样例一致时，`装货地 = 友谊`，`收货地 = 新登江丰`
- `公司名+过磅单标题版式`：顶部为 `浙江钧延贸易有限公司过磅单` 连在一起的标题，且整体打印版式与保利样例一致时，`装货地 = 友谊`，`收货地 = 保利`
- 若标题被遮挡、拍摄角度过斜、OCR 未识别出完整标题，或整体打印风格无法可靠判断，收货地留空或填 `暂定`，备注写 `钢坯收货地待核`
- 手写字只作为辅助参考，正式判断以顶部标题文字和整体打印版式为主
- 判断优先级：先按单据标题 / 版式特征判断，再套用批次特殊规则，最后使用普通默认规则
- 批次特殊规则：`5.25新登江丰` 批次为 `装货地 = 友谊`，`收货地 = 新登江丰`
- 普通默认规则：默认及 `5.25` 批次为 `装货地 = 友谊`，`收货地 = 保利`

其他货物：

- `装货地 = 暂定`
- `收货地 = 暂定`

### 6.7 备注

备注只写真实需要核对的内容。

| 情景 | 备注 |
|------|------|
| 车号不清楚 | `缺少车号` / `车号模糊` |
| 日期不清楚 | `日期模糊` |
| 日期依据冲突 | `日期冲突需核对` |
| 重量不清楚 | `重量模糊` |
| 重量位置冲突 | `重量冲突需核对` |
| 品名待确认 | `品名待核` |
| 钢坯收货地无法判断 | `钢坯收货地待核` |
| 驾驶员未匹配 | `未匹配到驾驶员` |
| 一图多单 | `一图多单` |
| 多图一单 | `多图合并` |
| 重复图片已剔除 | `重复N张已合并` |
| 原图缺失 | `原图缺失` |

禁止事项：

- 不使用自动阈值类备注 `重量异常,需核对`。
- 不把关键问题只放在辅助页。
- 不为了凑完整而猜测车号、重量、日期或货物名称。

## 7. 重复图片判定规则

### 7.1 总原则

重复剔除必须谨慎。

宁可保留重复，也不能把不同单据误删。

### 7.2 可以判定重复的情况

满足以下任一条件，才可以从主表剔除重复项，并写入 `重复图片` 页：

1. 原图文件哈希完全一致。
2. OCR 和原图核对后确认是同一张单据，且同时满足：
   - 单据编号一致
   - 完整毛重时间 / 皮重时间一致，或至少一个完整关键时间一致
   - 车号一致
   - 结算重量一致
   - 货物名称一致

### 7.3 不得判定重复的情况

以下情况不得作为重复剔除依据：

- 只有日期、车号、重量、货物名称相同。
- 只有重量相同。
- 单据编号不同。
- 完整毛重时间或皮重时间不同。
- 序号不同但其他结算字段巧合一致。
- OCR 缺少单号或缺少完整时间，无法证明是同一张单。

示例：

- `B202605050100` 和 `B202605050043` 即使车号、重量、货物相同，也不是重复，因为序号和完整时间不同。

### 7.4 重复图片页格式

页签名固定：

`重复图片`

固定两列：

| 重复图片 | 对应原图 |
|----------|----------|

主表保留 `对应原图` 那一条。

如果保留行需要承载重复说明，主表备注写：

`重复1张已合并`

## 8. 原图列要求

### 8.1 核心要求

`原图` 列默认写入外部原图链接，原始图片文件放在最终 Excel 旁边的同名 `_原图` 文件夹中。

必须做到：

- 一条主表记录对应一张原图。
- `原图` 列显示对应图片文件名，并写入可点击相对链接。
- 图片文件名必须包含序号、日期、车号、驾驶员和原始文件名等可读信息。
- 筛选、排序、移动行时，`原图` 链接单元格必须跟随对应行。
- 必须生成 `原图索引.csv`，记录生成时 Excel 行号、链接文件名、原图相对路径、原始文件名、日期、车号、驾驶员和重量。
- 交付文件中的原图链接必须是相对路径，不得依赖 `C:\Users\...` 这类本机绝对路径。
- 打开或查看图片时保持原图清晰度。

### 8.2 禁止做法

禁止为了缩小文件体积，把原图替换成二次压缩的小图。

不允许：

- 先生成低清缩略图再嵌入 Excel。
- 把原图压成模糊马赛克效果。
- 只写普通文件名但没有可点击链接和外部原图文件。
- Excel 与 `_原图` 文件夹脱离，导致交付后无法打开原图。

### 8.3 推荐实现方式

- 将原始 JPG/PNG 复制到 `运输录入_<日期范围>_原图` 文件夹。
- Excel `原图` 列写入相对链接，不把大图塞进工作簿。
- `原图` 列宽度可参考 `28`。
- 数据行高度可参考 `24`。
- 如确实需要单文件交付，可在软件中选择“格内嵌图（文件大）”作为可选模式。

### 8.4 筛选、排序和文件名关系

外部原图文件名在生成 Excel 时一次性确定。用户在 Excel 里筛选、排序或移动整行时，不要求、也不应该自动重命名外部图片文件。

必须保证：

- `原图` 单元格作为该行数据的一部分一起移动。
- 单元格里的相对链接始终指向该行对应的外部原图文件。
- `_原图` 文件夹和 Excel 一起复制、压缩或发送后，链接仍能打开。

如果用户在 Excel 生成后手工修改日期、车号、驾驶员等字段，外部图片文件名不会自动跟着改。需要文件名也按修改后的字段更新时，必须回到工具重新生成 Excel 和 `_原图` 文件夹。

说明：

- 默认外部链接模式用于解决 Excel 文件过大、打开和滚动卡顿、同事传输困难的问题。
- 不能为了轻量化牺牲原图清晰度和行级追溯关系。

## 10. 执行流程

### 10.1 批次预检查

- 确认本次只处理一个批次文件夹，不与历史批次混合。
- 软件启动后必须自动创建当天 `YYYY-MM-DD_保谊达做账表` 批次总目录，并在其中创建 `待处理图片` 与 `输出结果`。
- 清点原图数量，排除明显无关文件、测试文件、缓存文件。
- 检查是否已有 `paddle_ocr_raw.json`、`transport_parsed_records.json`、结构化 CSV 和最终 Excel。
- 已有完整中间文件时，先做验收和疑问复核，不直接重跑整批 OCR。
- 已有 `paddle_ocr_raw.json` 只有在当前批次每张图片都有非空 OCR 文本、没有 `error`、没有缺图、没有旧图记录时，才允许跳过 OCR。
- 非单据照片中如果包含有效单据区域，可保留并进入主表，同时保留原图便于核对。

### 10.2 OCR 获取

执行顺序固定：

1. 已有原始 OCR JSON 可用时，必须直接进入结构化解析和复核。
2. 原始 OCR JSON 缺失、损坏，或用户明确要求更换 OCR 引擎时，才重新 OCR。
3. OCR 引擎由执行工具按硬件和环境检测结果推荐，不由人工凭感觉切换。
4. 普通分享场景默认使用在线 GLM-OCR，以保证一个 exe 可直接运行。
5. 检测到强 NVIDIA GPU，且 PaddleOCR、PaddlePaddle GPU、CUDA、`paddle.utils.run_check()` 均通过时，才推荐切换本地 PaddleOCR。

一键出表前必须自动执行 OCR 预检，不允许把“检查 OCR 环境”做成只靠用户手动点击的摆设：

- 必须检查 `待处理图片` 中是否存在可处理图片。
- 必须检查已有 OCR JSON 是否完整可用；失败、空文本、缺图、混入旧图都必须自动重跑 OCR。
- 默认在线 GLM-OCR 时，必须检查 API Key 是否可用；缺 Key 必须弹窗提醒，停止生成 Excel。
- 选择本地 PaddleOCR 时，必须自动检查本地 Python、PaddleOCR、PaddlePaddle、CUDA / `run_check()`；环境不完整时不得强行切回在线 GLM-OCR，必须保持用户的本地选择，并进入本地 OCR 安装/配置流程。
- 本地 OCR 安装流程必须优先使用现有 Python；找不到 Python 时可通过 Windows `winget` 安装 Python 3.12；然后安装 PaddleOCR 和 PaddlePaddle。强 NVIDIA GPU 优先 GPU 包，Windows 50 系显卡按 PaddleOCR 官方 50-series wheel；GPU 未通过但 CPU Paddle 可用时允许用 CPU 本地模式，并明确提示速度较慢。
- OCR 有任何失败或返回空文本时，必须停止生成 Excel，不允许继续生成空白或错误表格。

本地 PaddleOCR 重跑命令（可选高级模式）：

```powershell
python scripts\run_paddle_ocr.py `
  --input-dir 5.31 `
  --output paddle_ocr_raw.json
```

执行要求：

- 整批 OCR 只跑一轮。
- 支持断点续跑时，已有 `file_name` 的图片应跳过。
- 重跑 OCR 会改变识别文本，必须重新走结构化解析、疑问复核和最终验收。

### 10.3 结构化解析

将 OCR JSON 转为主表数据、重复图片清单和疑问清单。

```powershell
python scripts\parse_transport_records.py `
  --input paddle_ocr_raw.json `
  --output-json transport_parsed_records.json `
  --output-csv 运输录入_结构化预览.csv `
  --duplicates-csv 重复图片.csv `
  --questions-csv 识别疑问_已回填主表.csv
```

结构化规则：

- 日期统一为 `YYYY/M/D`。
- 车号映射到标准车号。
- 驾驶员按固定表匹配。
- 重量统一换算为吨，保留两位小数。
- 品名按规则归一。
- 装货地、收货地按货物名称回填。
- 重复剔除必须按第 7 节规则执行，不得只用主表字段相同来剔除。

### 10.4 人工复核疑问行

复核对象：

- 日期冲突、日期模糊。
- 车号模糊、缺少车号、未匹配到驾驶员。
- 重量模糊、重量冲突。
- 品名待核。
- 钢坯收货地待核。
- 一图多单、多图合并、疑似重复。

处理规则：

- 能确认大部分字段的行必须进入主表。
- 无法确认的字段留空。
- 所有关键疑问必须写入主表 `备注`，不能只留在辅助 CSV。
- 复核时可对单张疑问图使用 GLM-OCR 或人工看原图，不反复整批 OCR。

### 10.5 Excel 生成与最终验收

生成 Excel：

```powershell
python scripts\build_transport_excel.py `
  --input-json transport_parsed_records.json `
  --output 运输录入_2026.5.12-2026.5.31.xlsx
```

生成后检查：

- 主表 `原图` 链接数量是否等于主表数据行数。
- `_原图` 文件夹是否存在，文件数是否等于主表数据行数。
- 点击 `原图` 列链接后，是否能打开对应清晰原图。
- `原图索引.csv` 是否能追溯生成时 Excel 行号、链接文件名、原图相对路径和原始文件名。
- `重复图片` 有内容时保留。
- 空的 `未匹配驾驶员` 删除。
- `识别疑问` 如果问题已进入主表备注则删除。
- 最终按第 11 节和第 15 节逐项验收。

## 13. Markdown 编码规范

执行文档统一保存为：

`UTF-8 with BOM`

原因：

- Windows 中文环境下，部分软件会把无 BOM 的 UTF-8 文件误判为 GBK。
- 误判后中文会显示为乱码。
- 使用 UTF-8 with BOM 后，常见编辑器能稳定识别中文编码。

VS Code 推荐设置：

```json
{
  "files.encoding": "utf8bom",
  "files.autoGuessEncoding": false
}
```

当前文件修复步骤：

1. 打开 Markdown 文件。
2. 点击 VS Code 右下角编码。
3. 选择 `Reopen with Encoding`。
4. 选择 `UTF-8`。
5. 确认中文显示正常。
6. 再点击右下角编码。
7. 选择 `Save with Encoding`。
8. 选择 `UTF-8 with BOM`。

## 14. OCR 配置和强制分层策略

### 14.1 OCR 引擎推荐规则

OCR 不固定死用某一个引擎，必须先按软件检测结果选择：

- 普通分享、无本地环境、无强 GPU、或使用者不懂环境配置时，默认在线 GLM-OCR。
- 本机有 5090 / 高显存 NVIDIA GPU，且 PaddleOCR、PaddlePaddle GPU、CUDA、`paddle.utils.run_check()` 全部通过时，优先本地 PaddleOCR PP-OCRv5。
- 有强 GPU 但 Paddle 环境不完整时，如果用户选择在线 OCR，当前批次先用在线 GLM-OCR；如果用户明确选择本地 OCR，软件必须先尝试补齐本地环境，不得静默改用在线 OCR。
- 已有可用 OCR JSON 时，不因引擎推荐变化而整批重跑。
- 软件界面必须使用中文引擎名和中文档位名，不得把 `glm`、`paddle`、`stable`、`fast`、`trt` 直接暴露给普通用户。

适用场景：

- 232 张左右的批量单据照片。
- 需要离线处理、断点续跑和保留原始 OCR JSON。
- 需要稳定拿到文本框、识别文本、置信度和空间位置供后续规则解析。

固定原则：

- 未完成抽样对比前，禁止为了追求“看起来更智能”而频繁切换 OCR 引擎。
- PaddleOCR 输出作为主解析链路，业务规则负责最终字段判断。
- OCR 只提供证据，不直接决定做账字段。

### 14.2 本地 5090 优化

5090 的优势主要是本地推理速度、TensorRT / FP16 和并发吞吐，不是自动提高业务判断正确率。

首次配置或重装运行环境时，必须先确认：

```powershell
python --version
python -m pip --version
nvidia-smi
```

Windows 50 系显卡必须先按 PaddleOCR 官方 50-series GPU wheel 说明判断适配方式；如果官方稳定版已支持当前驱动，再按 PaddlePaddle 官方 cu129 / cu130 安装路线执行。具体命令以官方页面最新说明为准。

安装后必须验证：

```powershell
python -c "import paddle; paddle.utils.run_check(); print(paddle.is_compiled_with_cuda())"
```

只有验证通过，才认为 GPU 环境可用于整批 OCR。

### 14.2.1 PaddleOCR 调用模板

整批 OCR 脚本必须只初始化一次 `PaddleOCR`，然后循环处理图片；禁止每张图片重新初始化 OCR 对象。

稳定档调用，作为正式处理基线：

```python
ocr = PaddleOCR(
    lang="ch",
    ocr_version="PP-OCRv5",
    device="gpu:0",
    use_doc_orientation_classify=True,
    use_doc_unwarping=True,
    use_textline_orientation=True,
    text_recognition_batch_size=8,
    textline_orientation_batch_size=8,
)
```

5090 加速档调用，小样本准确率不下降时用于整批：

```python
ocr = PaddleOCR(
    lang="ch",
    ocr_version="PP-OCRv5",
    device="gpu:0",
    enable_hpi=True,
    use_doc_orientation_classify=True,
    use_doc_unwarping=True,
    use_textline_orientation=True,
    text_recognition_batch_size=16,
    textline_orientation_batch_size=16,
)
```

TensorRT / FP16 条件档调用，只能在 TensorRT 环境验证通过后使用：

```python
ocr = PaddleOCR(
    lang="ch",
    ocr_version="PP-OCRv5",
    device="gpu:0",
    enable_hpi=True,
    use_tensorrt=True,
    precision="fp16",
    use_doc_orientation_classify=True,
    use_doc_unwarping=True,
    use_textline_orientation=True,
    text_recognition_batch_size=16,
    textline_orientation_batch_size=16,
)
```

参数调整规则：

- `enable_hpi=True`：小样本验证通过后启用高性能推理策略。
- `use_tensorrt=True`：TensorRT 环境验证通过后启用。
- `precision="fp16"`：小样本字段准确率不下降时启用。
- `batch_size`：按显存和稳定性逐步增加，不得一次性拉满。
- `text_recognition_batch_size`：优先从 `8` 提到 `16`；仍稳定且显存富余时再试 `24` 或 `32`。
- `textline_orientation_batch_size`：与识别 batch 同步调大；如显存波动或报错，先降回 `8`。
- `use_doc_unwarping=False`：只允许在样本确认图片不歪斜且字段准确率不下降时关闭；单据照片倾斜明显时必须保持开启。
- `text_rec_score_thresh`：不得随意调高，低置信度文本仍可能用于人工复核和备注判断。

以上加速项必须先小样本试跑，不得未经验证直接用于正式整批。

小样本试跑必须使用至少 20 张图片，覆盖钢坯、卷子板、钢渣、废铁类单据，以及日期冲突、车号模糊、重量模糊样例。对比通过标准：

- 主字段准确率不低于稳定档。
- 不新增漏识关键字段。
- 单张平均耗时明显下降。
- 输出结构仍能被后续解析脚本读取。

### 14.3 GLM-OCR 复核策略

GLM-OCR 仅限用于：

- 日期冲突、车号模糊、重量模糊等疑问行。
- 钢坯收货地待核、版式判断困难的单据。
- 表格、印章、复杂排版或普通 OCR 明显漏识的图片。
- OCR 引擎替换前的抽样对比。

GLM-OCR 可作为普通分享版默认整批 OCR，也可用于疑问行复核；在本地 5090 + Paddle GPU 环境完整时，是否替换为本地 PaddleOCR 必须先按 20 张样本比较字段准确率、漏识率、速度和成本。

样本对比必须记录：

- 字段准确率。
- 漏识和误识情况。
- 单张耗时和整批预计耗时。
- 本地算力或 API 成本。
- 对现有解析脚本结构的影响。

只有 GLM-OCR 在样本中明显优于 PaddleOCR，且输出能稳定转成 `paddle_ocr_raw.json` 等价结构时，才可替换整批 OCR 链路。

### 14.4 OCR 方案库

当前软件可直接调用的主方案：

- 在线智谱 GLM-OCR：普通分享默认方案，一个 exe 即可运行。
- 本地 PaddleOCR PP-OCRv5：本机 5090 / 高显存 GPU 环境完整后用于批量加速。

后续可接入或用于疑问行复核的方案：

- 通义 Qwen-OCR / Qwen-VL：优先作为在线备选和中文复杂票据复核方案。
- PaddleOCR PP-StructureV3 / PP-ChatOCRv4：用于表格、版式和关键信息抽取增强。
- 本地 Qwen2.5-VL / GOT-OCR2 / Surya：用于本地疑问行复核、抽样对比和复杂版式探索，不默认整批全量跑。
- Mistral OCR：用于 PDF / 文档解析场景的在线备选。

方案库规则：

- 新方案接入前，必须先抽样至少 20 张，比较字段准确率、漏识率、速度、成本和输出结构兼容性。
- 不允许默认多引擎全量重复跑整批图片；默认流程是主引擎整批、疑问行复核、人工确认。
- 任何新模型输出都必须经过业务规则解析和疑问标记，不得直接让模型猜做账字段。

### 14.5 密钥规则

- `ZHIPU_API_KEY` 只能保存在软件输入框、本机 `.env` 或系统环境变量中。
- 不得把真实 API Key 写入 Markdown、脚本、JSON、CSV、Excel 或公开仓库。
- 如果 API Key 泄露、失效或更换，只更新本机环境，不在执行文档中记录明文。

### 14.6 OCR 已知处理注意事项

- OCR 返回如出现中文乱码，应先做编码还原后再解析字段。
- `卷子板` 可能被误识别为 `电子板`、`番子板` 等，需结合原图和版式复核后按规则归一。
- `钢渣` 可能被误识别为 `钾渣`、`锅渣`、`铆渣`、`钠渣` 等，需结合原图和上下文复核后处理。
- 日期、单号年份、车号中间符号可能出现误读，必须结合毛重时间、皮重时间、编号和原图核对。
- 对结算字段不得猜测补全；无法确认时留空并写入主表备注。

## 15. 外部原图链接规则

最终 Excel 默认不嵌入原始图片文件。`原图` 列必须写入外部原图文件名和相对链接，原图文件放在 Excel 同目录的 `_原图` 文件夹中。

固定执行要求：

- 原图仍使用原始 JPG / PNG 文件，不生成低清二次压缩图。
- 每条主表记录对应一个外部原图文件。
- `原图` 列单元格显示外部原图文件名，并写入相对链接。
- 链接文件名必须可读，至少包含序号、日期、车号、驾驶员和原始文件名。
- `原图` 链接单元格必须随所在行排序、筛选、剪切、复制和移动。
- 必须生成 `原图索引.csv`，便于脱离 Excel 时核对图片来源；索引内必须包含原图相对路径，不写本机绝对路径作为交付依赖。

指定实现方式：

- 创建 `运输录入_<日期范围>_原图` 文件夹。
- 将每行对应原图复制到该文件夹。
- Excel `原图` 列写入 `external:` 相对链接或等效相对文件链接。
- `原图` 列宽度可参考 `28`。
- 数据行高度可参考 `24`。
- 如业务明确要求单文件交付，才允许选择“格内嵌图（文件大）”；该模式可使用 XlsxWriter `worksheet.embed_image()` 或等效 Excel Picture in Cell / richData 结构。

禁止做法：

- 不允许只写普通文本文件名但没有链接。
- 不允许生成低清压缩图替代原图。
- 不允许 Excel 与 `_原图` 文件夹分离交付。
- 不允许为了缩小 Excel 体积把原图压成模糊缩略图。
- 不允许链接数量、索引数量或外部原图文件数量与主表数据行数不一致。

最终检查追加项：

- Excel 是否能快速打开、滚动不卡顿。
- `原图` 列是否为可点击链接。
- 点击链接是否能打开清晰原图。
- 筛选、排序、移动行后，`原图` 链接是否仍对应该行。
- `_原图` 文件夹和 `原图索引.csv` 是否随 Excel 一起保留。
